# Story 015: VCS preference discoverability and override

**Status:** complete  
**Created:** 2026-05-14
**Last accessed:** 2026-05-15  
**Completed:** 2026-05-15

---

## Goal
Make the `vcs_preference` setting discoverable, documented, and changeable without hand-editing JSON. Currently `vcs_preference` in `.context/settings/project.json` is the only way to force a specific VCS when multiple are present, but no command writes it and no documentation mentions it. Auto-mode also silently prioritises Fossil over Git, which can surprise users who expected Git tooling.

## Verification
A user with both `.git` and `.fslckout` in the same project can run `/vcs-settings` and choose `Git/JJ` or `Fossil`, or run `/vcs-settings <auto|git|jj|fossil>` directly, and the footer immediately switches modes. If the selected tool is missing, Vazir prompts whether the user wants to install it; if the tool exists but the repo is not yet initialized for that mode, Vazir prompts before initializing it. The `project.json` file is updated and the preference persists across sessions. The setting is documented in a `.context/settings/README.md` file generated by `/vazir-init`.

## Completion Summary
Implemented VCS preference discoverability and override for story-015. The primary command is now `/vcs-settings`. `/vcs-settings` supports both direct args (`/vcs-settings <auto|git|jj|fossil>`) and a no-arg picker flow (`Auto`, `Git/JJ`, `Fossil`, `Cancel`). When the requested tool is missing, Vazir prompts whether the user wants to install it and then shows install guidance; when the tool exists but the repo is not initialized for that mode, Vazir now prompts before running the same initialization flow used by `/vazir-init` (`git init`, `jj git init --colocate`, or Fossil init/open). The command writes both `vcs_preference` and `active_vcs_mode` to `project.json` and refreshes the footer immediately, including fixing `auto` so it rewrites `active_vcs_mode` instead of leaving stale mode state behind.

Updated `resolvePreferredVcsKind` in `vazir-tracker/index.ts` to check `vcs_preference` as an explicit override layer before falling back to `active_vcs_mode` or auto-detection. Added `computeAutoDetectedVcsKind` to determine what auto-detection would choose, and wired `syncAndPublishVcs` to pass an `isOverridden` flag to `setVcsFlags`. Updated `branchLabel` in `chrome.ts` to append `*` when the resolved VCS kind differs from auto-detection. Added `SETTINGS_README_TEMPLATE` and wrote `.context/settings/README.md` during `/vazir-init`, then updated that README text to describe `/vcs-settings` and the picker flow. Added a `/vazir-init` re-run path that detects an existing setup and offers "Reconfigure VCS preference" vs "Full re-bootstrap". Added `/vcs-settings` to the `VAZIR_COMMAND_HELP` list. Added a `runVcsPreferenceOverrideScenario` to `validate-vazir-status-chrome.mts` to cover command-driven override behavior and footer rendering.
