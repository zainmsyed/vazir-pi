## Findings
### Finding 1
- Severity: medium
- Category: bug
- Summary: The advertised approval flow is not reliable across two separate user turns, so a user can follow the block message literally and still remain blocked.
- Evidence: `inspectVcsToolGuard()` in `.pi/extensions/vazir-tracker/vcs.ts` only authorizes a blocked action when `userInputHasVcsApproval(lastUserPrompt, existing.token)` is true for the current `lastUserPrompt`. Both extensions update `lastUserPrompt` only from the latest non-command input text. The block message says, `To approve exactly this action, send: VCS_APPROVE <token>. Then ask Vazir to retry the same command unchanged.` If the user sends the approval phrase in one message and asks to retry in the next message, the second turn overwrites `lastUserPrompt`, so the stored approval token is no longer present when the retried tool call is checked.
- Recommendation: Persist approval acknowledgement separately from the free-form last prompt, so the exact-token approval survives into the immediately following retry turn for the same fingerprinted action.
- Rule candidate: When a safety approval flow instructs the user to approve in one turn and retry in another, store the approval state independently from the latest natural-language prompt.

### Finding 2
- Severity: medium
- Category: test-gap
- Summary: There is no checked-in validation for the actual runtime `tool_call` guard path in `vazir-tracker`, so regressions in blocking or approval consumption would not be caught.
- Evidence: Story-017 changed the live interception path in `.pi/extensions/vazir-tracker/index.ts` and `.pi/extensions/vazir-tracker/vcs.ts`, but the only validation added in this area is `scripts/validate-vazir-vcs-safety-policy.mts`, which exercises helper functions and prompt assembly. It does not instantiate the tracker extension, emit `tool_call`, or assert that blocked calls return `{ block: true }`, notify the UI, and then allow a correctly approved retry.
- Recommendation: Add a tracker-focused validation harness scenario that loads the extension, emits blocked and approved `tool_call` events for `bash` and protected `write`/`edit`, and verifies both the block response and the approval-token retry behavior.
- Rule candidate: Add extension-level validation whenever new safety logic is wired into a live event interception path, not just helper-level unit checks.
