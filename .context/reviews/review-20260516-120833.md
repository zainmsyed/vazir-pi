# Story-005 Completion Review

## Summary

Reviewed the restored learned-rule closeout flow (`startLearnedRuleCloseout` → `finishLearnedRuleCloseout`) and its integration points with VCS commit behavior, the `agent_end` handler, and review closeout. The architecture is sound and the closeout path now correctly funnels through the learned-rule draft mechanism before finalizing. Found several bugs in edge-case handling, dead code from story-007 that was restored but not wired, a missing validation test, and a workflow concern about unnecessary I/O during skip paths.

## Findings

### 1. `updateRuleConfidence` imported but never called — dead code from story-007
- **Severity:** low  
- **Category:** dead-code  
- **Summary:** `updateRuleConfidence` is imported into `index.ts` (line 122) and exported from `helpers.ts`, but no call site exists in the current codebase. It was restored as part of the story-006/007 port but is not wired into any handler.  
- **Evidence:** `grep -n "updateRuleConfidence(" .pi/extensions/vazir-context/index.ts` returns zero matches.  
- **Recommendation:** Either wire it into the `/consolidate` handler as a story-007 follow-up, or remove the import from `index.ts` until story-007 is actively worked.  
- **Rule candidate:** Do not import or export functions that have no call site; if a helper is restored from an old branch, verify wiring before merging.

### 2. `promptLearnedRulePromotion` uses inconsistent terminology between text and select UIs
- **Severity:** low  
- **Category:** bug  
- **Summary:** The `ctx.ui.input` branch prompts the user to "save both rules" and "skip", while the `ctx.ui.select` branch uses "Promote all" and "Skip all". A user switching between TUI modes could be confused by the mismatch between "save" and "promote".  
- **Evidence:** `.pi/extensions/vazir-context/index.ts` lines 1041–1042: `Reply with \`both\` to save both rules, \`skip\` to save none...` vs. line 1064: `"Promote all"` and `"Skip all"`.  
- **Recommendation:** Align both branches to use "promote" consistently (e.g., "Reply with `both` to promote both rules, `skip` to promote none").  
- **Rule candidate:** When a flow offers both text-input and select-input paths, keep the action verbs identical across both UX modes.

### 3. `readLearnedRuleCloseoutDraft` silently drops candidates with non-lowercase confidence
- **Severity:** medium  
- **Category:** bug  
- **Summary:** The parser only accepts the exact strings `"high"`, `"medium"`, and `"low"` for confidence. If an LLM outputs `"High"` or `"HIGH"`, the candidate is filtered out with no warning, leaving the user with fewer candidates than the agent actually wrote.  
- **Evidence:** `.pi/extensions/vazir-context/helpers.ts` lines 2689–2697: strict `===` checks against lowercase strings.  
- **Recommendation:** Normalize confidence with `.toString().toLowerCase().trim()` before matching.  
- **Rule candidate:** Normalize enum-like string fields to lowercase before validation so minor casing variations from LLM output do not silently discard data.

### 4. `readLearnedRuleCloseoutDraft` can crash on `"null"` JSON input
- **Severity:** medium  
- **Category:** bug  
- **Summary:** After `JSON.parse`, the code casts the result as `Partial<LearnedRuleCloseoutDraft>` and immediately accesses `parsed.note`. If the file contains the literal string `"null"`, `JSON.parse` returns JavaScript `null`, and `null.note` throws `TypeError: Cannot read properties of null`.  
- **Evidence:** `.pi/extensions/vazir-context/helpers.ts` line 2686: `const parsed = JSON.parse(content) as Partial<LearnedRuleCloseoutDraft>;` followed by `typeof parsed.note`.  
- **Recommendation:** Add a runtime guard: `if (!parsed || typeof parsed !== "object") return { kind: "invalid", error: "JSON root is not an object" };`.  
- **Rule candidate:** Always guard `JSON.parse` results with a runtime type check before property access; `as` casts do not protect against malformed input at runtime.

### 5. No validation test for learned-rule closeout draft parsing edge cases
- **Severity:** medium  
- **Category:** test-gap  
- **Summary:** The `validate-vazir-complete-story.mts` script tests the happy path and the legacy candidates-file fallback, but there is no test for malformed JSON, empty files, missing `candidates` arrays, or candidates with invalid confidence values.  
- **Evidence:** Searching `scripts/validate-vazir-complete-story.mts` for `invalid`, `malformed`, or `JSON.parse` yields no draft-parsing edge-case tests.  
- **Recommendation:** Add a helper-level unit test or script assertion that exercises `readLearnedRuleCloseoutDraft` against: empty string, `"null"`, missing `candidates` field, and uppercase confidence values.  
- **Rule candidate:** Every data parser that consumes LLM-written files should have at least one edge-case test covering malformed, null-root, and missing-field inputs.

### 6. Legacy mini-consolidate candidates file fallback adds complexity without deprecation timeline
- **Severity:** low  
- **Category:** simplification  
- **Summary:** `finishLearnedRuleCloseout` contains a fallback branch that reads from the old `story-NNN-candidates.md` format. The fallback is only reachable when the JSON draft is missing or invalid. `parseMiniConsolidateCandidates` and `miniConsolidateCandidatesPath` are imported solely for this transitional path.  
- **Evidence:** `.pi/extensions/vazir-context/index.ts` lines 1104–1122: legacy fallback block.  
- **Recommendation:** Schedule removal of the legacy fallback once the new JSON draft format has been stable for a few stories, or add a `TODO` comment with a target date.  
- **Rule candidate:** When replacing a file format, add a TODO with a removal date for the backward-compatibility shim so it does not linger indefinitely.

### 7. `organizeLearnedRules` runs during skip paths, doing unnecessary I/O
- **Severity:** low  
- **Category:** workflow  
- **Summary:** In `finishLearnedRuleCloseout`, `organizeLearnedRules(ctx.cwd)` is called unconditionally after promotion, even when the user selects "Skip all" (no rules promoted). The function scans every source story of every unkinded rule to determine failure vs success kind.  
- **Evidence:** `.pi/extensions/vazir-context/index.ts` line 1165: `organizeLearnedRules(ctx.cwd);` is outside the `if (selection === "skip")` branch.  
- **Recommendation:** Move `applyLocalRuleDedupe` and `organizeLearnedRules` inside the `else` branch so they only run when rules were actually promoted.  
- **Rule candidate:** Do not trigger global rule reorganization I/O when the user explicitly chose to skip promotion.

### 8. `agent_end` falls through to story readiness check when review is not yet complete
- **Severity:** medium  
- **Category:** bug  
- **Summary:** In the `agent_end` handler, if `processCompleteStoryReviewCloseout` returns `false` (review not yet complete and not resumable), the code falls through to `assessStoryCompletionReadiness` and can re-prompt `promptReadyCloseout`. This means a user who already chose "Start review" may be asked again "Would you like to close or review?" before the review has finished writing.  
- **Evidence:** `.pi/extensions/vazir-context/index.ts` lines 1549–1560: after `handledReview` is false, the code continues to readiness checks instead of returning early.  
- **Recommendation:** If `reviewFilePath` exists but the review is not complete and we are not resuming, return early from `agent_end` rather than falling through.  
- **Rule candidate:** When a pending review is blocking closeout, the `agent_end` handler should yield rather than re-prompting the user with the same closeout menu.

## Recommended Fixes

- [x] medium — Guard `readLearnedRuleCloseoutDraft` against `null` JSON root and normalize confidence casing before validation.  
- [x] medium — Add edge-case validation tests for `readLearnedRuleCloseoutDraft` (null root, empty file, uppercase confidence, missing candidates field).  
- [x] medium — Return early from `agent_end` when a review is pending but not yet complete, instead of falling through to readiness re-prompt.  
- [x] low — Remove `updateRuleConfidence` import from `index.ts` (story-007 will re-add when wiring).  
- [x] low — Align terminology in `promptLearnedRulePromotion` text-input branch to use "promote" instead of "save".  
- [x] low — Move `organizeLearnedRules` and `applyLocalRuleDedupe` inside the non-skip branch of `finishLearnedRuleCloseout`.  
- [x] low — Add a dated TODO for removing the legacy mini-consolidate candidates file fallback.

## Completion Summary

The restored learned-rule closeout architecture is functionally correct and passes all validation scenarios including close-and-commit. The primary concerns are edge-case robustness in the JSON draft parser, a missing test gap, and a few workflow/paper-cut issues. None of the findings are release-blocking for story-005, but the parser guards and the `agent_end` fall-through should be fixed before the story is considered fully hardened.
