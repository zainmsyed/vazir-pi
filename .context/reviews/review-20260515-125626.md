## Findings
### Finding 1
- Severity: medium
- Category: test-gap
- Summary: Story-016's new VCS safety helpers are not covered by an automated validation, so regressions in protected-target detection or approval-gated command matching would currently ship unnoticed.
- Evidence: The implementation adds new behavior in `.pi/lib/vazir-helpers.ts` (`isProtectedVcsTarget`, `protectedVcsTargetsInText`, `approvalGatedVcsOperation`, `buildVcsSafetyGuidanceText`, `buildDefaultSystemRulesMarkdown`), but no validation script under `scripts/` was added or updated for story-016. Verification was done with ad hoc `node --experimental-strip-types` shell snippets rather than a checked-in harness scenario.
- Recommendation: Add a targeted validation that asserts positive and negative cases for the protected-path detector, destructive command matcher, and system-guidance text assembly across Git, JJ, and Fossil examples.
- Rule candidate: When adding shared policy helpers that parse commands or paths, add a checked-in validation covering representative positive and negative cases.

### Finding 2
- Severity: low
- Category: simplification
- Summary: The new VCS safety rules are injected twice for freshly bootstrapped projects, which adds avoidable prompt bloat.
- Evidence: `.pi/lib/vazir-helpers.ts` now includes the safety rules in `buildDefaultSystemRulesMarkdown()`, and `.pi/extensions/vazir-context/index.ts` also unconditionally appends `buildVcsSafetyGuidanceText()` during `before_agent_start`. A project created after this change will therefore get the same policy once from `.context/memory/system.md` and again from the extra `[VCS Safety Policy]` block.
- Recommendation: Reuse the shared helper while avoiding duplicate prompt content, e.g. only inject the extra safety block when the current `system.md` does not already contain the policy text.
- Rule candidate: Prefer shared prompt-policy builders, but avoid injecting the same policy both from persisted memory and from an additional runtime block.
