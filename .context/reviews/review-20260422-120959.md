## Findings
### Finding 1
- Severity: high
- Category: bug
- Summary: Footer rendering omits the active story slug (UI regression / failing validation).
- Evidence: Running the project's validation suite fails in validate-vazir-status-chrome.mts with an assertion error "footer did not include the story slug". The failing test expects the footer to include the active story id (e.g. "story-003"). Example failure (abbreviated):

  Error: footer did not include the story slug
    at file:///.../scripts/validate-vazir-status-chrome.mts:327:3

  The assertion in the test checks: `footerLines[1]?.includes("story-003")`.

  Investigation shows sessionFooterLine (in .pi/extensions/vazir-tracker/chrome.ts) builds the footer left-hand segments but does not include the story label variable (summary?.slug). The code constructs leftSegments and joins them but never injects the story slug into that list.
- Recommendation: Update the footer assembly to include the active story slug in the left segment (consistent with the documented UX). For example, insert the story label after the repo name, e.g.:

    const leftSegments = [
      paint(`◈ ${repoNameLabel(cwd)}`, "accent", true),
      paint(summary?.slug ?? "no active story", "text"),
      branchWithStatus,
      ...
    ];

  Run the full validations (npm test / scripts/run-validations.mts) after the change and update any related docs (docs/vazir-tui-ux.md) if the wording/format changes.
- Rule candidate: Footer should always include the active story slug for easy orientation.

### Finding 2
- Severity: low
- Category: dead-code
- Summary: Large editor/backup files (.bak) are present under .pi/extensions and appear to be committed backups rather than active source files.
- Evidence: Repository contains:
  - .pi/extensions/vazir-context.ts.bak (approx. 62,913 bytes)
  - .pi/extensions/vazir-tracker.ts.bak (approx. 68,255 bytes)

  These files are redundant with the active typescript sources at .pi/extensions/vazir-context/index.ts and .pi/extensions/vazir-tracker/*.ts and can confuse tooling (live-reload watchers, code readers) and increase repository size.
- Recommendation: Remove or move these .bak files out of the repository history (git rm and commit), or relocate them to a single archive folder outside source tree. Add a .gitignore entry to prevent future commits of editor backup artifacts (e.g. *.bak, *~). Ensure the live-reload watcher ignores backup files (it already uses an IGNORE_FILE_PATTERN but confirm it excludes .bak at the watcher pattern level).
- Rule candidate: Avoid committing editor or backup files (e.g. .bak, ~, .tmp).

### Finding 3
- Severity: medium
- Category: dead-code / scope drift
- Summary: A large repository backup bundle (repo-backup.bundle) is present in the workspace root; this bloats the repository and is likely unnecessary in source control.
- Evidence: repo-backup.bundle is present at repo root (~795 KB according to repository listing) and .gitignore contains an entry to ignore repo-backup.bundle, suggesting it should not be kept in the tracked history.
- Recommendation: Remove the bundle from the repository history (git rm) and store off-repo (external backup storage). If a history snapshot is required in-tree for CI/debugging, consider using Git LFS or a separate release artifact rather than committing a bundle into main repo history.
- Rule candidate: Do not commit large binary backups into the main repository; use external storage or LFS.

### Finding 4
- Severity: low
- Category: workflow
- Summary: Guidance mismatch on whether .context should be treated as persistent/project brain vs ignored artefact.
- Evidence: AGENTS.md states: "Keep .context/ as the persistent project brain". However .gitignore includes an entry for .context/ (the repo currently has .context files committed, but the presence of .gitignore suggests ambiguity about whether team should track those files).
- Recommendation: Clarify the intended workflow: either (A) commit .context artifacts that must be shared and remove .context/ from .gitignore, or (B) continue to keep .context locally ignored and update AGENTS.md to indicate that .context is intentionally workspace-local. Update docs and onboarding notes accordingly.
- Rule candidate: Make the team policy explicit about which .context subpaths (if any) are authoritative and tracked.

### Finding 5
- Severity: low
- Category: simplification / test-gap
- Summary: Ambient TypeScript declaration files under types/ use permissive "any" types which reduces the ability of local validation scripts to catch API changes or misuse of the pi runtime surface.
- Evidence: types/pi-runtime-ambient.d.ts and types/node-runtime-ambient.d.ts declare many module exports and APIs as `any`, and the validator harness stubs modules during tests. This is pragmatic for the harness but means subtle signature regressions may be missed.
- Recommendation: Tighten the ambient declarations where practical (at least for the parts of the runtime used by the extensions). Consider importing or referencing the real SDK types or incrementally improving the ambient .d.ts files so the validation scripts can catch signature regressions earlier.
