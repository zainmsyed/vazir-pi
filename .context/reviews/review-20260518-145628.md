## Findings

### Finding 1
- Severity: medium
- Category: bug
- Summary: Review instruction template has duplicate item number 13, which can confuse the agent when generating review files.
- Evidence: `.pi/extensions/vazir-context/helpers.ts` lines 2424–2425: both lines are prefixed with `` `13. `` — one for "Review focus" and one for "Story".
- Recommendation: Renumber the second 13 to 14 and bump the existing 14 to 15.
- Rule candidate: When generating numbered requirement lists in instruction builders, scan for duplicate ordinals before committing.

### Finding 2
- Severity: high
- Category: bug
- Summary: VCS safety guard for direct `write`/`edit` tool calls only checks the `path` property, missing tools that use `filePath` instead.
- Evidence: `.pi/extensions/vazir-tracker/vcs.ts` line 60: `const rawPath = (input as { path?: unknown }).path;`. The `toolPathFromInput` helper in `chrome.ts` already falls back from `path` to `filePath`, but the guard does not.
- Recommendation: Update `directToolApprovalRequirement` to also inspect `filePath`, matching the fallback logic in `toolPathFromInput`.
- Rule candidate: When validating tool input for security, always check every documented alias of a target field (`path` / `filePath`), not just the primary one.

### Finding 3
- Severity: medium
- Category: bug
- Summary: `/design` command hardcodes `story-003` in provenance comments regardless of the active story.
- Evidence: `.pi/extensions/vazir-context/index.ts` lines 2825, 2835, and 2859 all contain literal `<!-- source: story-003 -->`. If a user edits the design system from a different story, the provenance is wrong.
- Recommendation: Resolve the active story label at runtime (via `findActiveStory`) and interpolate it into the comment.
- Rule candidate: Never hardcode story labels or temporal identifiers in reusable command handlers; always derive them from runtime state.

### Finding 4
- Severity: medium
- Category: bug
- Summary: `resolvePreferredVcsKind` can return `"jj"` even when JJ is not actually active, misleading the footer and checkpoint flows.
- Evidence: `.pi/extensions/vazir-tracker/index.ts` line 306: `if (vcsPreference === "jj" && hasGitRepo) return "jj"; // jj requires git`. This ignores the `useJJ` flag, so if the user sets preference to JJ but never initialized it, Vazir advertises JJ mode without working checkpoints.
- Recommendation: Change the condition to `vcsPreference === "jj" && useJJ` so the returned kind matches actual runtime state.
- Rule candidate: Always verify runtime capability flags before returning a mode or state label; preference settings are intent, not proof of readiness.

### Finding 5
- Severity: medium
- Category: bug
- Summary: Edit-stream tracking uses a FIFO queue keyed only by tool name, so parallel `write`/`edit` calls that finish out of order attribute the wrong file to the "done" event.
- Evidence: `.pi/extensions/vazir-tracker/chrome.ts` lines 698–707: `pushPendingEditCall` appends to an array and `claimPendingEditCall` claims the first matching toolName. If two `write` calls are in flight and the second returns first, the stream records the first call's file as completed.
- Recommendation: Include a unique call ID (timestamp + counter) in the pending call and match on it in `tool_result` rather than by toolName alone.
- Rule candidate: When tracking async tool lifecycle, correlate start and end events with a unique call identifier, not just the tool name.

### Finding 6
- Severity: medium
- Category: bug
- Summary: Story status-change approval detector uses substring `includes()`, producing false positives on negated phrases like "do not mark this complete."
- Evidence: `.pi/extensions/vazir-context/helpers.ts` lines 892–950: `userExplicitlyApprovedStatusChange` checks `normalized.includes(phrase)` for phrases like `"mark this complete"` and `"close story"`. A user saying "do not mark this complete" or "don't close story yet" would incorrectly match.
- Recommendation: Add boundary checks — require the phrase to be preceded by a word boundary or start of string, and reject matches preceded by negation words ("not", "don't", "never").
- Rule candidate: Substring-based approval detection must guard against negation prefixes; use word-boundary regexes or explicit allow-list tokens rather than naive `includes()`.

### Finding 7
- Severity: low
- Category: bug
- Summary: Fossil VCS sync does not handle `RENAMED` status, so renamed files are silently omitted from the footer changed-files display.
- Evidence: `.pi/extensions/vazir-tracker/vcs.ts` lines ~495–510: `syncFromFossil` matches `EDITED`, `UPDATED_BY_MERGE`, `MISSING`, `ADDED`, and `DELETED` from `fossil changes` output, but has no branch for `RENAMED`.
- Recommendation: Add a `RENAMED` matcher that extracts the new filename and inserts it with an appropriate status code (e.g. `R`).
- Rule candidate: When parsing VCS status output, include rename/move states explicitly; they are common and silently dropping them creates UI gaps.

### Finding 8
- Severity: high
- Category: test-gap
- Summary: No automated tests exist for core logic paths (story completion, review closeout, VCS guardrails, helper functions). The only test file is a trivial `node -e` smoke test.
- Evidence: `test_exec.js` is the sole test file. Complex flows like `finishLearnedRuleCloseout`, `inspectVcsToolGuard`, `syncFromFossil`, and `userExplicitlyApprovedStatusChange` have no unit coverage.
- Recommendation: Add a minimal test runner (Node built-in or `node:test`) and seed it with tests for the most critical helpers: VCS guard, status-change approval, edit-stream tracking, and story frontmatter parsing.
- Rule candidate: Every state-machine or guardrail helper that blocks or auto-approves user-facing actions must have at least one direct unit test exercising both the allow and deny paths.

### Finding 9
- Severity: low
- Category: simplification
- Summary: Misleading indentation in `/vazir-init` makes several bootstrap calls look like they are outside the `if (!reconfigureOnly)` guard.
- Evidence: `.pi/extensions/vazir-context/index.ts` lines 1959–1963: `ensureDir(settingsDir(cwd))`, `ensureIntakeStructure`, `ensureReviewStructure`, `ensureArchiveStructure`, and `ensureDir(path.join(cwd, ".context", "checkpoints"))` are indented at the same level as the `if` statement, even though the closing brace is far below. JavaScript braces make this syntactically correct, but the visual structure is deceptive.
- Recommendation: Re-indent lines 1959–2078 to 8 spaces so they align with the first two `ensureDir` calls inside the block.
