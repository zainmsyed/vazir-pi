# Story-006 Completion Review

## Summary

Reviewed the restored Fallow recurrence flow in `vazir-context/helpers.ts`, its review-closeout wiring in `vazir-context/index.ts`, and the new validation coverage added for review closeout and signal-source handling. The helper logic and review-closeout append/dedupe flow are in place, but I found one live prompt bug in a direct integration point and one gap where the new validation only checks instruction text rather than proving real signal consumption.

## Findings

### 1. `buildMiniConsolidateInstruction` still emits a literal `${path.basename(reviewFilePath)}` instead of the actual review filename
- **Severity:** medium  
- **Category:** bug  
- **Summary:** The legacy mini-consolidate instruction builder still uses a plain quoted string for the review-file line, so when a review file exists it tells the agent to read `.context/reviews/${path.basename(reviewFilePath)}` literally instead of the real path. This is a live regression in a direct story-006 integration point because the instruction now claims to use review and `[fallow]` signal together.  
- **Evidence:** `.pi/extensions/vazir-context/helpers.ts:2536` contains `"- Read the review file at .context/reviews/${path.basename(reviewFilePath)}."` rather than a template literal.  
- **Recommendation:** Convert that line to a template literal and add a small assertion that the built instruction contains the actual review basename when `reviewFilePath` is provided.  
- **Rule candidate:** When restoring or updating prompt builders, add one assertion that interpolated file paths render as concrete paths rather than raw `${...}` placeholders.

### 2. The new `[fallow]` signal-source validation only proves prompt wording, not actual candidate/promotion consumption
- **Severity:** medium  
- **Category:** test-gap  
- **Summary:** Story-006 claims `/consolidate` and mini-consolidate now recognize `[fallow]` complaints as valid signal sources, but the new validation only checks that instruction strings include `[fallow]` and `.context/complaints-log.md`. It does not prove that a complaints-log-only `[fallow]` signal can actually influence candidate generation or promotion in a real closeout/consolidation path.  
- **Evidence:** `scripts/validate-vazir-fallow-signal-sources.mts` only asserts `.includes("[fallow]")`, `.includes(".context/complaints-log.md")`, and story-label text; it does not seed `complaints-log.md` and exercise a real consolidate or closeout consumer.  
- **Recommendation:** Add one higher-level validation that seeds `.context/complaints-log.md` with a story-specific `[fallow]` entry and verifies the downstream path consumes that signal — either by exercising a deterministic helper/parser or by asserting a generated candidate/promotion outcome, not just prompt text.  
- **Rule candidate:** When adding a new signal source to an instruction-driven workflow, validate one real consumption path end-to-end instead of only checking that the prompt mentions the new source.

## Recommended Fixes

- [x] medium — Fix `buildMiniConsolidateInstruction` so review file paths are interpolated correctly and add a regression assertion for the rendered basename.  
- [x] medium — Add an end-to-end validation that seeds `[fallow]` complaints-log input and proves a real consolidate or closeout consumer uses it, rather than only checking prompt text.
