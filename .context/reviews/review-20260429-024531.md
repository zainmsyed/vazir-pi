### Finding 1 — Widespread duplication in validation scripts
- Severity: medium
- Category: simplification
- Summary: Large, repeated test-harness code exists across multiple validation scripts (ensureStubModule, createProject, makePi/makeCtx, writeStory helpers), increasing maintenance cost and the risk of inconsistent fixes.
- Evidence: nearly identical helper blocks appear in:
  - scripts/validate-vazir-init.mts
  - scripts/validate-vazir-review-loop.mts
  - scripts/validate-vazir-complete-story.mts
  - scripts/validate-vazir-story-status-guard.mts
  - scripts/validate-vazir-fix-routing.mts
  - scripts/validate-vazir-edits-stream.mts
  - scripts/validate-vazir-checkpoint-labels.mts
  (see repeated functions ensureStubModule, createProject, makePi, makeCtx in those files)
- Recommendation: Factor the shared test harness into a single importable module (e.g., scripts/_harness.mts or scripts/lib/validation-harness.mts). - Rule candidate: Promote a rule requiring shared test harness utilities for repository validation scripts (yes)

### Finding 4 — Fragile core areas: .context persistence, change-tracking and fix workflow, extension APIs, live-reload
- Severity: high
- Category: workflow
- Summary: AGENTS.md and the codebase both identify core areas that are brittle: .context persistence, change-tracking/fix workflow, extension APIs, and hot-reload. These are central to agent correctness and recovery; regressions here cause data loss or incorrect agent behavior.
- Evidence: AGENTS.md explicitly lists these paths as "Fragile areas"; extensions implementing these features are located under .pi/extensions (vazir-context, vazir-tracker, vazir-live-reload). Large extension files and extensive filesystem manipulation appear in scripts and extensions.
- Recommendation: Prioritize integration tests that exercise .context write/read cycles, the /fix workflow, and extension reload behavior. Add atomic write safeguards (tempfile+rename), explicit backups when mutating .context files, and clear error handling/retries for extension reload failures. Run end-to-end scenarios in CI to detect regressions early.
- Rule candidate: Require an automated integration test for any change touching .context persistence or extension APIs (yes)

### Finding 5 — Missing CI enforcement for validations and static analysis
- Severity: medium
- Category: test-gap
- Summary: There's no repository workflow that runs repository validations or Fallow static analysis automatically on push/PR. This increases the chance that the "fallow audit" findings will not be visible to contributors before merges.
- Evidence: package.json defines a local "test" script that runs scripts/run-validations.mts, but there is no .github/workflows CI job present to run npm test on PRs. The repository contains a .github directory with non-ci files only.
- Recommendation: Add a CI workflow (GitHub Actions or equivalent) that runs npm test and the Fallow static analysis on pull requests. Fail the job on any validation or fallow failures. Add a periodic job for full-run fallow to capture wider findings.
- Rule candidate: Require CI to run the repo validation suite and static analysis before merge (yes)

### Finding 6 — Test runner uses Node experimental flag (--experimental-strip-types)
- Severity: low
- Category: simplification
- Summary: The validation runner invokes node with --experimental-strip-types, an unstable flag that may change across Node versions and break tests unexpectedly.
- Evidence: scripts/run-validations.mts calls childProcess.execFileSync(process.execPath, ["--experimental-strip-types", scriptPath], ...).
- Recommendation: Replace the experimental flag usage with one of the following: precompile TypeScript files in a step, use ts-node or a stable ts runner, or document the required Node version and pin it in CI. Prefer a stable approach for CI and contributors.
